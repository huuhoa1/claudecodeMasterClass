# Spec for signup-firebase-auth

branch: claude/feature/signup-firebase-auth
figma_component (if used): N/A

## Summary

Wire the existing signup form at `/signup` to Firebase Authentication so that submitting it creates a real user account. On successful signup, a randomly generated codename (three words joined in PascalCase, one from each of three distinct word sets) is set as the user's Firebase `displayName`. A document is then written to a Firestore `users` collection storing the user's `id` and `codename` — but not their email. Only the Firebase Web SDK is used; no server-side code.

## Functional Requirements

- The signup form submits credentials to Firebase Auth using email/password sign-up
- On successful account creation, a codename is generated by randomly selecting one word from each of three unique word sets and joining them in PascalCase (e.g. `SwiftCrimsonFox`)
- The generated codename is set as the Firebase user's `displayName` via `updateProfile`
- A document is created in a Firestore `users` collection with the fields `id` (the Firebase UID) and `codename` (the generated display name) — the user's email is never stored
- If signup fails (e.g. email already in use, weak password), a meaningful error state is surfaced to the user
- After successful signup the user remains signed in (Firebase handles this automatically)

## Possible Edge Cases

- Firebase signup succeeds but `updateProfile` or Firestore write fails — the account exists but is incomplete; this should not leave the UI in a broken state
- Codename collision in Firestore is possible but acceptable at this scale — no uniqueness enforcement required for now
- The word sets must have enough variety to keep codenames feeling distinct; each set should contain at least 20 words
- The form should be disabled / show a loading state while the async signup flow is in progress to prevent double-submission

## Acceptance Criteria

- Submitting the signup form with a valid email and password creates a Firebase Auth user
- The newly created user has a `displayName` matching the generated PascalCase codename
- A document exists in `users/{uid}` containing `{ id, codename }` with no email field
- Submitting with an already-registered email shows an appropriate error message
- The form cannot be submitted twice while a signup is in progress
- After successful signup, `useUser()` returns the signed-in user with the correct `displayName`

## Open Questions

- Should the user be redirected after signup, and if so to which route?
- Should the three word sets be defined inline in a utility file, or loaded from a separate data file?

## Testing Guidelines

Create a test file(s) in the ./tests folder for the new feature, and create meaningful tests for the following cases, without going too heavy:

- Submitting the form calls Firebase `createUserWithEmailAndPassword` with the correct arguments
- On success, `updateProfile` is called with a non-empty PascalCase display name
- On success, a Firestore document is written to the `users` collection with `id` and `codename` but no `email`
- On failure, an error message is displayed to the user
- The codename generator always returns a non-empty string in PascalCase from the three word sets
